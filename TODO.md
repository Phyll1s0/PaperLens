# PaperLens TODO

## Found Problems

- [x] 当前运行服务可能还是旧进程：已增加 health schema v2、运行状态面板和命令行队列快照；如果仍提示旧 schema，需要重启当前服务。
- [ ] 扫描版 PDF 目前仍偏“提示型”：能识别并给 OCR 建议，但还不能在 PaperLens 内自动 OCR、回填文本、继续走解析队列。
- [ ] 公网部署风险还没有兜住：如果直接部署到服务器，上传、论文库、模型调用和导出接口缺少登录或访问令牌保护。
- [ ] OpenAI-compatible Provider 的代理链路不够稳定：普通 `fetch` 对 HTTP/SOCKS 代理支持有限，Kimi/DeepSeek 在本机代理、Docker、服务器环境下表现可能不同。
- [ ] Claude Code + Kimi Code Key 仍依赖本机环境：需要 `claude` CLI、隔离配置、代理和预算状态都正确，别人复现成本偏高。
- [ ] 长任务缺少硬性资源保护：大 PDF、批量重建视觉结构、批量补跑会消耗 CPU、磁盘和模型预算，需要更明确的限流与取消策略。
- [ ] 现在主要依赖 JSON 文件存储：并发写入、文件体积增长、异常退出时存在数据损坏或恢复困难的风险。
- [ ] 视觉结构仍是启发式识别：复杂双栏、多图组合、跨页图、公式/代码混排时仍可能裁剪过大或误分类。
- [ ] AI 分段缺少人工编辑闭环：分段错了以后会影响翻译、讲解、导出，目前还不能在 UI 中合并/拆分并只重跑受影响段落。
- [ ] 导出结果缺少质量校验：Markdown/Word 导出还需要预览、引用完整性检查、图片缺失诊断。
- [ ] 测试覆盖不足：核心路径仍以 `node --check`、health 和手工跑为主，缺少 PDF fixture、Job Queue、导出和 Provider 诊断回归测试。
- [ ] README 与实际能力有漂移：最近新增的服务版本提示、OCR 指引、视觉维护、速度仪表盘需要同步进文档。
